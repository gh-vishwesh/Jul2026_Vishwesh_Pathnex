Day 13 - Advanced Automation with CloudFormation and CI/CD

🔹 Ansible - Multi-Node Nginx Setup
- name: Setup Nginx on multiple nodes						----> Title of the automation: Setup Nginx on multiple nodes
  hosts: all									----> run all servers 
  become: yes									----> become administration 
  tasks:									----> Start writing test 
    - name: Install nginx							----> just a title 
      yum:									----> use yum package manager 
	name: nginx 								----> Install package named nginx
	state: present								----> ensure nginx exists, if already installed nothing happens and if not install it

    - name: Start nginx service							----> Task title
      service:									----> manage Linux services, equivalent to systemctl
	name: nginx								----> Target service nginx
	state: started								----> equivalent to systemctl start nginx
	enabled: yes								----> means start automatically after reboot

🔹 Terraform - Create IAM Users and Policies
resource "aws_iam_user" "example_user" {					----> Create an AWS resource. "aws_iam_user" - resource type, "example_user" - Terraform name 
  name = "vishwesh_user"							----> Actual IAM user inside AWS
}

resource "aws_iam_policy" "example_policy" {					----> Create an IAM policy
  name		= "vishwesh-policy"						----> Policy name
  description	= "Policy for Vishwesh application"				----> Description shown inside AWS Console.
  policy	= jsonencode({							----> AWS expects policy in JSON. Terraform converts the object into JSON automatically.
    Version	= "2012-10-17"							----> AWS policy language version [Almost every policy uses it.]
    Statement	= [								----> policy rules start
      {
	Action = "s3:ListBucket"						----> allowed action [list bucket contents]
	Effect = "Allow"							----> Permission granted could also be Deny
	Resource = "arn:aws:s3:::vishwesh-bucket"				----> Permission applies to only to vishwesh-bucket, not every bucket
      },
    ]
  })
}

🔹Kubernetes - Deploy Nginx with ConfigMap and Secrets
apiVersion: v1									----> API version
kind: ConfigMap									----> object type, ConfigMap stores configuration not passwords
metadata:									----> object information.
  name: nginx-config								----> ConfigMap name
data:										----> actual configurations begins
  nginx.conf: |									----> Store entire nginx.conf file.
    server {									----> nginx server block
      listen 80;								----> listen on HTTP port 80
      server_name localhost;							----> server name
      location / {								----> Homepage route
	root /usr/share/nginx/html;						----> website files stored here
	index index.html index.htm;						----> Default page.
      }
    }

---
apiVersion: v1
kind: Secret									---> secret stores sensitive data
metadata:
  name: nginx-secret
type: Opaque									---> Generic secret
data:
  password: dmlzaHdlc2ggcGFzcw==						----> This is Base64 encoded for vishwesh pass

---
apiVersion: apps/v1
kind: Deployment								----> Deploy pods
metadata:
  name: vishwesh-nginx 								----> Deployment name
spec:
  replicas: 2									----> Create 2 pods High availability
  selector: 									----> Deployment identifies Pods using Labels
    matchLabels:								----> match labels, app=vishwesh-nginx
      app: vishwesh-nginx
  template:									----> Blueprint of pod
    metadata:
      labels:									----> attach labels
	app: vishwesh-nginx
    spec:
      containers:								----> Container list
- name: nginx									----> Container name
	  image: nginx								----> Pull nginx image
	  volumeMounts: 							----> Mount storage into container
	    - mountPath: /etc/nginx/nginx.conf					----> Replace nginx.conf inside container
	      name: nginx-config
	      subPath: nginx.conf						----> Only mount one file instead of the whole ConfigMap directory.
      volumes:									----> define storage
	- name: nginx-config
	  configMap:								----> Volume comes from configMap
	    name: nginx-config
	- name: nginx-secret
	  secret:								----> volume comes from secret
	    secretName: nginx-secret

🔹Jenkinsfile - Multi-Branch Pipeline with Docker Build
pipeline {									----> Pipeline starts
    agent any									----> Run on any available Jenkin agents.
  stages {									----> Pipeline stages begin
	stage('Build') {							----> Build stage.
	    steps {
		script {
		    docker.build('vishwesh-app')				----> Creates docker image, Equivalent to docker build -t vishwesh-app .
		}
	    }
	}
	stage('Deploy') {							----> Deployment stage
	   steps {
		script {
		    kubectl apply -f kubernetes/deployment.yaml			----> Deploy applications to Kubernetes
		}
	    }
	}
    }
}

🔹GitLab CI/CD - Advanced Deployment with Helm
stages:										----> Pipeline stages
  - build									----> Compile/build
  - push									----> Push docker image
  - deploy

build:
  stage: build
  script: 
    - docker build -t vishwesh-web-app .					----> Create docker image

push:
  stage: push
  script:
    - docker login -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD"		----> login to docker registry, GitLab username: CI_REGISTRY_USER and Password/token: CI_REGISTRY_PASSWORD

deploy:
  stage: deploy
  script:
    - helm repo add vishwesh https://charts.vishwesh.com			----> Add helm chart repository
    - helm install vishwesh-nginx-ingress					----> Install application using Helm

[Think of Helm as the package manager for Kubernetes, similar to how apt installs software on Ubuntu.]

🔹 Docker
# COPY vs ADD
FROM ubuntu									----> use ubuntu as base image
COPY vishwesh-file.txt /opt/vishwesh/files/					---> copy one file, [quivalent to Local machine -> Docker Image]
ADD vishwesh.tar.gz /opt/vishwesh/extracted/					----> ADD copies like COPY, if the source is a local .tar archive, docker automatically extracts it.


Remember it as like:
👨‍💻 Developer → Git Push → Jenkins/GitLab CI → Docker Build → Push Image to Registry → Terraform (Provision AWS Resources) → Ansible (Configure Servers) → Kubernetes (Deploy Application) → 🚀 Application Running
